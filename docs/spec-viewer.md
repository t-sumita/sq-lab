# sq-lab V1 仕様書

策定: 2026-07-05(S1) / 版: v0.1.0

## 目的

sq-lab は [squash-lab](https://github.com/t-sumita/squash-lab)(private)が
出力するプレイヤー分析データ(`result.json`)を閲覧するための、公開された
静的ビューアである。コーチが選手にデータを見せながら指導するための道具、
というユーザー像を想定している。

squash-lab 自体は動画解析を行うPCバッチツールであり、公開を前提としない。
sq-lab はその出力(JSON)だけを扱う、完全に分離された別リポジトリ・別サイト
として運用する。

## 設計原則

1. **読み込んだ試合データは端末から出ない**: 読み込んだ result.json は
   ブラウザ内メモリ上でのみ処理する。drag&drop またはファイル選択で読み込み、
   その内容を外部へ送信するコードは書かない。デモデータは file:// でも
   確実に動くよう JS 変数として同梱し、fetchを使わない。
   なお v0.5.0 で追加した Subutomo 共通バッジ(`assets/subutomo-badge.js`)は
   同一オリジンの静的な他サイト一覧(`config/subutomo-sites.json`)を
   遅延fetchするが、これは**ユーザーの試合データを一切含まない静的ファイル**
   であり、上記原則(試合データの不送信)とは矛盾しない。「外部通信ゼロ」
   ではなく「**あなたの試合データを外部へ送信しない**」が本サイトの不変条件
   である。
2. **サイト全体が外部の第三者サーバーへ何も送信しない**: Ko-fi/GitHub
   Sponsors/Feedbackのリンクはユーザーが能動的にクリックして遷移する先の
   外部サイトであり、本サイトのコードが自動的に通信するものではない。
3. **実データを入れない**: リポジトリに同梱するのは完全な合成デモデータ
   (架空のプレイヤー名・人工的な位置と指標値)のみ。実際の試合動画・
   実際の result.json・the Analyzer の内部設計文書は混入させない。
4. **vanilla JS + Canvas、ビルドステップなし**: フレームワーク・バンドラ・
   npm 依存を使わない。`<script type="module">` は `file://` で CORS
   ブロックされるため使わず、`window.SQLab` 名前空間に吊るした通常の
   script で構成する。ヒートマップ・リプレイ(S2/S3)は Canvas で描画する。
5. **対象ユーザー像**: 「コーチが選手に見せながら教える道具」。専門用語は
   最小限にし、モバイル(スマホ縦画面)でも問題なく使えることを優先する。

## V1 スコープ

- **S1**: リポジトリ骨格、JSON読込+スキーマ検証、試合情報ヘッダ、
  P1/P2サマリーカード(距離/Work Rate/T支配率/カバレッジ/平均速度)、

- **S3**: リプレイ(positions の時系列をアニメーション再生、
  再生/シーク/倍速、軌跡のフェード表示、欠損の誠実な可視化)
- **v0.4.0 "PR1"**: プレリリース仕上げ(P0読込修正・テンプレート準拠・
  guide.html・プレイヤー名編集)
- **v0.5.0(本リリース)**: 初回表示のデモ自動表示化、spctrl-site準拠
  (フッタ/バッジ/右上ⓘ)、JA表示の自然な日本語化、カード化・8pxグリッドの
  デザイン原則適用、分析タブの再構成(Heatmapグループ化・Movement Replay
  への改名)、リプレイのトレイル3択(Off/5s/All)・16倍速、
  キャッシュバスティング(`?v=`)
- **S4**: 公開(GitHub Pages + Cloudflare DNS で `sq-lab.subutomo.dev`、
  最終チェック)

## 対応スキーマ

sq-lab は squash-lab の `docs/spec-schema.md` が定義する `result.json`
スキーマ **v0.2** にのみ対応する。`schema_version` フィールドがこれと
一致しない場合は、対応バージョンと読み込まれたバージョンを併記した
エラーを表示し、読込を中止する(v0.1 を含め、v0.2 以外は全て不可)。

sq-lab が実際に参照するトップレベルフィールド(S1時点):

| フィールド | 用途 |
|---|---|
| `schema_version` | バージョン検証 |
| `video.file` / `video.analyzed_fps` | 試合情報ヘッダ |
| `analyzed_segments` | 主カメラ区間の合計時間(試合情報ヘッダ) |
| `generator.name` / `generator.version` | 試合情報ヘッダ |
| `metrics_params.t_zone` | T支配率カードの定義注記(a_m/b_m/offset_m を動的表示) |
| `unassigned_track_ids` | 未割当断片がある場合の注記表示 |
| `players[].id` / `players[].metrics` | サマリーカード本体 |
| `players[].positions` | S2(ヒートマップの正規化には未使用)/S3(リプレイの再生・軌跡・欠損判定)で使用 |
| `players[].metrics.heatmap` | S2ヒートマップで使用 |

squash-lab 側でスキーマに加算的変更(新フィールド追加)があった場合、
sq-lab 側は無視して問題ない設計とする(未知フィールドは読み飛ばす)。
破壊的変更(既存フィールドの削除・型変更)があった場合のみ
`schema_version` を上げ、sq-lab 側も追随して `SUPPORTED_SCHEMA`
(js/config.js)を更新する。

## 画面構成(v0.5.0時点)

1. **ヘッダ**: ロゴ(`assets/subu-pic-header.png`、未配置でも崩れない)+
   サイト名「Squash Laboratory」(クリックで`index.html`へ)、バージョン表示、
   右上ⓘ(guide.htmlへの導線)、言語切替ボタン(表示中言語の2文字のみ)
2. **初期表示**: 起動直後に**デモデータの分析結果を自動表示**する
   (v0.5.0で変更。以前は読込前のイントロ画面が既定だった)。上部に
   「これはサンプルデータです」の帯+「ファイルを読み込む」ボタンを表示し、
   押すと読込UI(drag&dropゾーン+ファイル選択、エラー表示欄)を展開する。
   ページ全体がドロップ領域を兼ねる(読込UIを開いていなくても良い)
3. **結果パネル**(実データ読込後はサンプルデータ表示帯が消える):
   - 試合情報カード(動画ファイル名・解析フレームレート・主カメラ区間の
     合計時間・生成ツール名/バージョン、未割当がある場合はⓘのみ表示)
   - P1(赤系)/P2(青系)サマリーカード(薄い枠線カード、指標ごとにⓘ)
   - 分析セクション(Heatmap: Side by Side/Overlay のタブグループ +
     Movement Replay タブ)
4. **フッタ**: copyright + Ko-fi + Sponsors + Feedback の3リンク
   (spctrl-site準拠。「Analyzed by ...」等のタグラインはv0.5.0で撤去し、
   データ非送信の説明はguide.html冒頭に集約した)
5. **左下バッジ**: Subutomo共通バッジ(他サイト一覧+ロゴ+copyright)

言語切替は `localStorage` に保存され、リロード後も維持される。指標名の
うち Work Rate / T-Dominance / Coverage 等の英字表記は国際的に定着した
ものを除き、v0.5.0でJA表示を自然な日本語(移動距離・T支配率・
コートカバー率等)に改めた。モード名(Side by Side / Overlay /
Movement Replay)は指標名と同様、言語によらず英語表記で統一している。

## 画面構成(S2/S3: 「分析」セクション)

結果パネルのサマリーカードの下に「分析」(Analysis)セクションを追加する。
S2まではこのセクションのタブ・T-zoneトグルは `heatmap.js` が自前で持って
いたが、S3で Replay をヒートマップと同列のタブとして追加するにあたり、
タブ・T-zoneトグル・リサイズ監視は `js/views.js` に一元化した。
`heatmap.js` / `replay.js` はそれぞれ「与えられたコンテナに描画する」
低レベルAPI(`Heatmap.renderSideBySide` / `Heatmap.renderOverlay` /
`Replay.render`)のみを提供する構成に整理してある。

- **「Heatmap: Side by Side | Overlay」グループ + 「Movement Replay」**
  の構成でタブ表示モードを切替(v0.5.0でグループ化・Replayを改名。
  モード名は指標名と同様に英語表記で統一)
- **Side by Side**: P1・P2のコートを横並び(狭幅では縦積み)。各コートの
  上にプレイヤー名ラベルと0〜共通最大値(秒)の凡例を表示
- **Overlay**: 1枚のコートにP1(赤)/P2(青)を加法合成で重畳し、
  両者が重なる領域を紫系として可視化。凡例は「P1のみ/両方/P2のみ」の
  色見本
- **Movement Replay**: court.js のコート上にP1(赤丸)/P2(青丸)を現在時刻の
  位置に描画し、再生/一時停止・シークバー・倍速(1x/2x/4x/8x/16x)・
  軌跡3択(Off/5s/All、v0.5.0で追加)を提供する。All は全軌跡を残す
  モードで、確定済み区間をオフスクリーンcanvasに累積描画し(巻き戻し時のみ
  作り直す)毎フレーム全体を引き直さないことで60fpsを維持する。
  隣接サンプルの間隔が0.3秒以下なら線形補間し、それを超える
  (=実際の追跡欠損)場合は丸を表示しない(最後の位置に留め置かない)。
  シークバー下のデータ充足ストリップで、P1/P2それぞれのデータ有無(赤/青の帯)と
  主カメラ外の時間帯(グレー)を可視化する
- **T-zoneトグル**: metrics_params.t_zone(a_m/b_m/offset_m)から算出した
  楕円を点線で表示/非表示切替。「T-Dominanceの定義」をサマリーカードの
  注記と対応させて視覚的に伝える。3タブ共通で利用可能
- ヒートマップの正規化は **両プレイヤー共通の最大値** を用いる(片方だけ
  濃く見えて比較を誤らせないため)

## デモデータの二重管理について

`demo/demo-match.json`(人間が読む参照用・ダウンロード用)と
`js/demo-data.js`(`window.SQLab.DEMO_DATA` として同梱、fetchを使わず
file:// でも確実に動くよう埋め込んだコピー)は**同一内容**を保つ必要が
ある。デモデータを更新する際は必ず両ファイルを同時に更新すること
(生成スクリプトで一括生成し、後から手動で個別編集しないこと)。

デモのプレイヤー表示名は言語対応のため `display_name` にマーカー文字列
(`__DEMO_A__` / `__DEMO_B__`)を格納し、`js/playernames.js` がi18n経由で
実際の表示名(EN: "Player A (demo)" 等)に解決する。実データの
`display_name` はこのマーカーと衝突しない限り、そのまま表示される。

## 検証工程(v0.5.0で恒常ルール化)

コミット前の file:// E2E に加え、**push承認後・完了報告前に
`https://sq-lab.subutomo.dev` への実サイトE2E**(キャッシュ無効化リロード
相当)を実施し、バージョン表示・フッター文言・i18nキー未解決ゼロを
確認すること。file:// では再現しないキャッシュ関連の不具合(CDNキャッシュ、
ブラウザキャッシュ)は実サイトでしか検出できないため。

S3では、リプレイの欠損可視化を実際に確認できるよう、Demo A(P1)の
positions に意図的な約2秒の欠損区間(t=50.0〜52.0付近)を含めている。
これも生成スクリプト側でフィルタしたうえで両ファイルへ同時反映した。
